# Проектная работа "Веб-ларек"

Стек: HTML, SCSS, TS, Webpack

Структура проекта:
- src/ — исходные файлы проекта
- src/components/ — папка с JS компонентами
- src/components/base/ — папка с базовым кодом

Важные файлы:
- src/pages/index.html — HTML-файл главной страницы
- src/types/index.ts — файл с типами
- src/index.ts — точка входа приложения
- src/scss/styles.scss — корневой файл стилей
- src/utils/constants.ts — файл с константами
- src/utils/utils.ts — файл с утилитами

## Данные и типы данных, исполльзуемые в приложении

Карточка

```
export interface ICard {
    id: string;
    description: string;
    image: string;
    title: string;
    category: string;
    price: number;
}
```
Пользователь

```
export interface  IUser {
    payment: string;
    email: string;
    phone: string;
    address: string;
    total: number;
    items: ICard[];
}
```
Интерфейс для модели данных карточек

```
export interface ICardsData {
    cards: ICard[];
    prewiev: string | null;
	deleteCard(cardId: string, payload: Function | null): void;
	getCard(cardId: string): ICard;
}
```
Открытая карточка

```
export type TCardInfo = Pick<ICard, 'description' | 'image' | 'title' | 'category' | 'price'>;
```
Отображение карточки в карзине

```
export type TCardBasket = Pick<ICard, 'title' | 'price'>;
```
Для ввода адреса пользователя

```
export type TUserAdres = Pick<IUser, 'address'>;
```
Для ввода почты и телефона пользователя

```
export type TUserContact = Pick<IUser, 'email' | 'phone'>;
```
Для отображения списаных средств пользователя

```
export type TUserTotal = Pick<IUser, 'total'>;
```


## Архитектура приложения

Код приложения разделен на слои согласно парадигме MVP: 
- слой представления, отвечает за отображение данных на странице, 
- слой данных, отвечает за хранение и изменение данных
- презентер, отвечает за связь представления и данных.

### Базовый код

#### Класс Api
Содержит в себе базовую логику отправки запросов. В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов.
Методы: 
- `get` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер
- `post` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.

#### Класс EventEmitter
Брокер событий позволяет отправлять события и подписываться на события, происходящие в системе. Класс используется в презентере для обработки событий и в слоях приложения для генерации событий.  
Основные методы, реализуемые классом описаны интерфейсом `IEvents`:
- `on` - подписка на событие
- `emit` - инициализация события
- `trigger` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие   


### Слой данных

#### Класс CardsData
Класс отвечает за хранение и логику работы с данными карточек созданных пользователями.\
В полях класса хранятся следующие данные:
- _cards: ICard[] - массив объектов карточек
- _preview: string | null - id карточки, выбранной для просмотра в модальной окне.

Так же класс предоставляет набор методов для взаимодействия с этими данными.
- deleteCard(cardId: string, payload: Function | null): void; - Удаляет карточку из корзины
- getCard(cardId: string): ICard; - Возвращает карточку по ее id

#### Класс UserData
Класс отвечает за хранение и логику работы с данными текущего пользователя.\
Конструктор класса принимает инстант брокера событий\
В полях класса хранятся следующие данные:
- payment: string - В сети ли пользователь
- email: string - Емаил полльзователя
- phone: string - Номер телефона пользователя
- address: string - Адрес пользователя
- total: number - Общая сумма покупок
- items: ICard[] - Массив с карточками товаров выбранных пользователем
- events: IEvents - экземпляр класса `EventEmitter` для инициации событий при изменении данных.

Так же класс предоставляет набор методов для взаимодействия с этими данными.
- getUserInfo(): TUserTotal - Возвращает общую сумму покупок
- setUserInfo(userData: IUser): void - сохраняет данные пользователя в классе
- checkUserValidation(data: Record<keyof TUserContact, string> | Record<keyof TUserAdres, string>): boolean - проверяет объект с данными пользователя на валидность

### Классы представления
Все классы представления отвечают за отображение внутри контейнера (DOM-элемент) передаваемых в них данных.

#### Класс Modal
Реализует модальное окно. С подстановкой в него данных через темплейт.Так же предоставляет методы `open` и `close` для управления отображением модального окна. Устанавливает слушатели на клавиатуру, для закрытия модального окна по Esc, на клик в оверлей и кнопку-крестик для закрытия попапа.\
Поля:
- isOpen: boolean - Указывает, открыто ли модальное окно. Инициализируется как false.
- content: HTMLElement | null - Хранит текущий контент, отображаемый в модальном окне. Изначально null.
- modalElement: HTMLElement - Элемент HTML, представляющий модальное окно, который передается в конструктор.

Методы:
- constructor(modalElement: HTMLElement) - Конструктор класса Modal, принимает элемент HTML, представляющий модальное окно.
- open(content: HTMLElement): void - Открывает модальное окно с заданным контентом. Если окно уже открыто, выводит предупреждение в консоль.
- close(): void - Закрывает модальное окно. Если окно уже закрыто, выводит предупреждение в консоль.
- toggle(content: HTMLElement): void - Переключает состояние модального окна. Если окно открыто, закрывает его. Если закрыто, открывает с заданным контентом.
- isOpened(): boolean - Возвращает true, если модальное окно открыто, иначе false.

#### Класс ProductModal 
Расширяет класс Modal. Преднозначен для реализации модального окна с выыбранной карточкой товара.\
Методы:
- openWithProduct(product: { id: string, name: string, description: string, price: number }, addToCartCallback: (productId: string) => void): void - Открывает модальное окно с информацией о выбранном товаре. Принимает объект товара и функцию обратного вызова для добавления товара в корзину.
- createProductContent(product: { id: string, name: string, description: string, price: number }, addToCartCallback: (productId: string) => void): HTMLElement - Создает и возвращает контент для модального окна с информацией о товаре и кнопкой для добавления в корзину.

#### Класс CartModal
Расширяет класс Modal. Преднозначен для реализации модального окна с корзиной.\
Поля:
- cartItems: { id: string, name: string, quantity: number }[] - Массив объектов, представляющих элементы корзины. Каждый элемент содержит идентификатор и название.
Методы:
- constructor(modalElement: HTMLElement, cartItems: { id: string, name: string, quantity: number }[]) - Конструктор класса CartModal, принимает элемент HTML, представляющий модальное окно, и массив элементов корзины.
- openWithCartItems(): void - Открывает модальное окно с элементами корзины. Генерирует контент на основе текущего состояния корзины и отображает его в модальном окне.
- createCartContent(): HTMLElement - Создает и возвращает контент для модального окна в виде списка элементов корзины. Каждый элемент отображается с кнопкой для удаления.
- removeItem(itemId: string): void - Удаляет элемент из корзины по его идентифик.

#### Класс FormInputModal
Расширяет класс Modal. Преднозначен для реализации модального окна с формой ввода.\
Методы:
- FormInputModal, принимает элемент HTML, представляющий модальное окно.
- openWithForm(fields: { label: string, name: string }[], onSubmit: (data: any) => void): void - Открывает модальное окно с формой ввода. 
- createFormContent(fields: { label: string, name: string }[], onSubmit: (data: any) => void): HTMLElement - Создает и возвращает контент для модального окна с формой ввода.

#### Класс ConfirmationModal 
Расширяет класс Modal. Преднозначен для реализации модального окна подтверждения.\
Методы:
- ConfirmationModal, принимает элемент HTML, представляющий модальное окно.
- openWithConfirmation(message: string, onConfirm: () => void): void - Открывает модальное окно с сообщением и кнопками для подтверждения действия. При клике на кнопку подтверждения вызывается переданная функция onConfirm и модальное окно закрывается.
- createConfirmationContent(message: string, onConfirm: () => void): HTMLElement - Создает и возвращает контент для модального окна с сообщением и кнопками.

#### Класс CardsContainer
Отвечает за отображение блока с карточками на главной странице. 

### Слой коммуникации

#### Класс AppApi
Принимает в конструктор экземпляр класса Api и предоставляет методы реализующие взаимодействие с бэкендом сервиса.

## Взаимодействие компонентов
Код, описывающий взаимодействие представления и данных между собой находится в файле `index.ts`, выполняющем роль презентера.\
Взаимодействие осуществляется за счет событий генерируемых с помощью брокера событий и обработчиков этих событий, описанных в `index.ts`\
В `index.ts` сначала создаются экземпляры всех необходимых классов, а затем настраивается обработка событий.

*Список всех событий, которые могут генерироваться в системе:*\
*События изменения данных (генерируются классами моделями данных)*
- `card:selected` - изменение открываемой в модальном окне карточки
- `card:previewClear` - необходима очистка данных выбранной для показа в модальном окне карточки

*События, возникающие при взаимодействии пользователя с интерфейсом (генерируются классами, отвечающими за представление)*
- `card:select` - выбор карточки для отображения в модальном окне
- `card:delete` - выбор карточки для удаления из корзины
- `email:input` - ввод почты пользователя
- `phone:input` - ввод номера телефона пользователя
- `address:input` - ввод адреса пользователя
- `address:submit` - сохранение адресса пользователя
- `emailPhone:submit` - сохранение номера телефона и почты пользователя
- `remove-card:submit` - событие, генерируемое при нажатии "Оплатить" в форме подтверждения
- `email:validation` - событие, сообщающее о необходимости валидации формы почты
- `phone:validation` - событие, сообщающее о необходимости валидации формы номера телефона
- `address:validation` - событие, сообщающее о необходимости валидации формы адресса
